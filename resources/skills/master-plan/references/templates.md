# Document templates — reference for the master-plan skill

Exact templates for every file the skill generates. `{{slots}}` are fill-ins;
everything else is emitted verbatim so every project gets the identical
structure. Budgets cited here are checked by `planning/tools/plan_check.py`
(FAIL blocks handoff; WARN is advisory) — the script is the source of truth
for the numbers.

Evidence keys used in comments: CutClean = 1.7MB PROGRESS.md / 345KB plan-part
collapse; TowDefender = the evolved convention that mostly held; Lantern Lock =
verification-table + facts-for-future-sessions checkpoint; DMGoblin = 2,088-char
inline ✅ essays + orphaned root plan; Schlorm = gates + step-ID commits.

---

## 1. PROGRESS.md (project root — THE dashboard, the only status doc)

```markdown
<!-- PLANDOC:PROGRESS v1 — generated by the master-plan skill.
     THE dashboard. The owner reads ONLY this file to know where the project stands.
     BUDGETS (checked by planning/tools/plan_check.py, run at every session close):
       file <=32,768 bytes and <=400 lines · every line <=120 chars (target 100)
       NEXT STEP block <=6 lines, pointer only — banned from restating the step spec
       Session log: last 3 entries only · 6 labeled fields · <=240 chars per field · <=16 lines/entry
       Amendments <=3 lines each (advisory) · Decisions <=20 live, one line each · Facts <=15 bullets
     Overflow rotates VERBATIM to planning/archive/. Spec never lives here (plans own it).
     Status never lives anywhere else. Narrative essays live in git commit bodies —
     why-this-state essays, collapsible details-tag blocks, and full-detail-in-…
     cross-links are check failures; point to commits as "see commit <sha7>". -->
# PROGRESS — {{Project Name}}
Updated: {{YYYY-MM-DD}} · Session S{{0000}} · Branch: {{branch}} · HEAD: {{sha7}}

## ▶ NEXT STEP
NEXT: {{P01-S01}} — {{step title}}
Plan: planning/plans/{{PLAN-01-slug}}.md
Note: {{<=2 sentences of orientation. Never restate the step spec.}}

## Dashboard — all plans
| Plan | Slug | Status | Done |
|---|---|---|---|
| PLAN-01 | {{slug}} | 🔵 ACTIVE | 0/{{N}} |
**All plans: 0/{{N}} steps complete.**
(Plan statuses: 🔵 ACTIVE · ⏳ QUEUED · ⏸️ PARKED → archive/PLAN-NN-PARKED.md · ✅ COMPLETE · ❌ ABANDONED)

## Board — PLAN-01 (active)
Legend: ⬜ todo · 🔧 in progress · ✅ done · ⏸️ parked · ❌ dropped (see amendment) · 👤⬜/👤🔧/👤✅ owner-gated
| Step | Title | M | Status | Done (date · sha7) |
|---|---|---|---|---|
| P01-S01 | {{title}} | M1 | ⬜ | |
| P01-S02 | {{title}} | M1 | ⬜ | |

## Completed plans
(one line per closed plan → full board in its closeout file)

## Blockers (open only — resolved blockers are deleted; history lives in the session archive)
- none

## Environment register
(the ONLY home for volatile env/deploy state: committed-not-pushed work, pending
migrations, env vars required before next push. Never restate any of it in session
entries, NEXT STEP, CLAUDE.md, or OWNER_TODO — manual rule; the checker cannot
detect paraphrased duplication.)
- none

## Facts future sessions need (curated <=15 bullets — edit in place, delete superseded facts)
- Gates: see planning/reference/GATES.md ({{gate name}}: `{{command}}`)
- {{environment fact / sync invariant ("change one, change all three") / pitfall}}

## Amendments — active plan (append-only; `- A-P<NN>-<##> (date, affects <steps>): …` <=3 lines; closed→closeouts)
(none)

## Decisions (one line each; max 20 live — oldest roll to the closeout/archive)
- D-001 ({{date}}): {{decision one-liner}}

## Session log (newest first; exactly the last 3 live here; older are in planning/archive/)
### S0000 — {{date}} — system initialized
- Did: master-plan INIT — planning system generated from {{draft path}}.
- Verified: plan_check.py exit 0.
- Decisions: none.
- Handoff: {{branch}} · {{sha7}} · green.
- Next: P01-S01.
- Notes: none.
```

Rules baked in:
- **Status cells are one legend entry** (owner-gated cells prefix 👤 to the state
  glyph). Dates/SHAs go in the Done column. Prose in a status cell is the
  CutClean 300-char cell rot — a check failure.
- **Flipping the board row IS marking done** (plus one session entry). There is
  no second place to record completion.
- The header `Updated:` line is refreshed every session close — the checker's
  staleness tripwire compares its HEAD sha against git HEAD.
- The session entry's six labels are exactly: `Did:` `Verified:` `Decisions:`
  `Handoff:` `Next:` `Notes:`.
- Unplanned work (hotfix, emergent pass) still gets a session entry with the
  step field reading `unplanned: <reason>`, and an Amendment if it touched spec.

## 2. SESSION_PROMPT.md (project root — static, THE single copy)

````markdown
<!-- PLANDOC:SESSION_PROMPT v1 — IMMUTABLE (hash recorded by plan_check.py).
     The ONLY copy of this prompt in the repo. Other docs LINK here, never embed it
     (an embedded copy is what drifted at CutClean). The moving part — which step is
     next — lives in PROGRESS.md ▶ NEXT STEP, never here. -->
# Start-of-session prompt — copy the fenced block below

Open a fresh Claude Code session in `{{project folder}}/` and paste this:

```
You are continuing the {{PROJECT NAME}} build ({{one line: what it is}}).
Multi-session build; you have a fresh context window.
One step is your entire job today.

1. Read PROGRESS.md — all of it (it is size-capped by design, so this is safe).
2. Find "## ▶ NEXT STEP". It names one step ID and one plan file. That single
   step is your whole job. Do NOT read whole plan files or planning/archive/.
   If it says PLAN COMPLETE or none, stop and tell the owner to run the
   master-plan skill.
3. In the named plan file, read ONLY your step's block (search "[<step-id>]")
   plus whatever its Context field cites, plus the previous step's Carryover
   (a plan's first step has none).
4. Verify green FIRST: run the gate named in the step's last Completion box
   (commands in planning/reference/GATES.md). If red, fixing it starts your job.
5. Implement EXACTLY that one step, satisfying every Completion-criteria box
   and its Verify command. Plan files are immutable — any deviation is a dated
   <=3-line Amendment entry in PROGRESS.md, never an edit to a plan file.
6. Close out, same session, in this order:
   a. git commit as "<step-id>: <title>" (narrative detail goes in the commit
      body — nowhere else). Push only if the step says deploy; set any new
      env vars FIRST, then verify the deploy.
   b. Update PROGRESS.md in one pass: refresh the header Updated line (date ·
      session · branch · HEAD), flip the board row (glyph + Done cell), update
      the Dashboard count, advance ▶ NEXT STEP (pointer only; after the plan's
      LAST step write "NEXT: PLAN COMPLETE — run the master-plan skill"),
      refresh Blockers/Environment register, prepend ONE 6-field session
      entry, and move the oldest entry past 3 into planning/archive/
      (shard + numbering rules: planning/reference/CONVENTIONS.md).
   c. Add any human-only work to OWNER_TODO.md ## Open, tagged with the step
      ID; if you set PLAN COMPLETE, add an Open item to run the master-plan
      skill to close out this plan.
   d. Run: python planning/tools/plan_check.py — it must exit 0 before you stop.
   Never hand off a red build or a failing doc check.

If the step overflows the session: reach a green sub-state, commit it, split
the step in PROGRESS.md (<id>a ✅ / NEXT: <id>b), log a one-line Amendment.
Never end a session red.

Begin by stating the step ID, its Objective verbatim, and your plan for it.
```
````

Design notes: static-with-external-pointer (CutClean's best idea); plan-file
agnostic — survives PLAN-05 without edits; gate-name indirection via GATES.md
survives stack changes; bounded reading list fixes TowDefender's 200KB boot tax.
Budget: generated file <=3.5KB (emitted verbatim from this template — never
append project material). The sentence "One step is your entire job today."
must stay on one physical line — the checker greps for it verbatim to enforce
the single-copy rule. No-git projects: replace step 6a's commit instruction
with "record the step ID + a one-line summary in the session entry" (see
SKILL.md STEP 1).

## 6. planning/reference/CONVENTIONS.md (generated once per project)

```markdown
<!-- PLANDOC:CONVENTIONS v1 — generated by the master-plan skill; refreshed on
     CHECK & REPAIR runs (the Project-specific additions section is preserved). -->
# Planning conventions — {{Project Name}}

## Authority order (highest wins)
code/repo state > plan file + its Amendments > PROGRESS.md > SESSION_PROMPT.md.
A doc may not name another doc as source of truth for status it itself carries.

## IDs (fixed grammar, never renumbered)
- Plans: PLAN-NN-<slug> (NN monotonic, zero-padded, never reused).
- Steps: P<NN>-S<KK>, flat within the plan; splits get letter suffixes (P01-S04a/b).
- Sessions: S<NNNN>. Amendments: A-P<NN>-<##>. Decisions: D-<###>.
- Commits for step work are prefixed "P02-S07: " — the drift-proof progress spine.
  (No-git projects: no commit spine — the board + session log are the only
  completion evidence; header slots read `Branch: none · HEAD: no-git`.)

## The four sources of truth (no overlap)
| File | Carries | Mutability |
|---|---|---|
| planning/plans/PLAN-NN-*.md | spec | immutable once LOCKED (hash-checked) |
| PROGRESS.md | state | updated every session, size-capped |
| SESSION_PROMPT.md | protocol | immutable (hash-checked) |
| OWNER_TODO.md | human actions | items move Open → Done |

## Session protocol (summary — the full text is SESSION_PROMPT.md, linked not copied)
One step per session · verify green first · deviations are Amendments ·
close-out updates PROGRESS.md + runs plan_check.py · never hand off red.

## Step-split procedure (step overflows the session)
Reach a green sub-state → commit → in PROGRESS.md board: <id>a ✅ (Done cell
filled) + new row <id>b ⬜ → NEXT: <id>b → one-line Amendment. Never end red.

## Session-log rotation (close-out)
- New session ID = newest hot entry + 1 (S<NNNN>, zero-padded, never reused).
- After prepending, move the oldest entry past 3 VERBATIM to the END of the
  highest-numbered planning/archive/SESSIONS-*.md shard (oldest first).
- A shard holds exactly 25 entries; when full, start the next range
  (e.g. SESSIONS-0026-0050.md). If planning/archive/ has no shard yet, create
  SESSIONS-0001-0025.md with the PLANDOC:ARCHIVE header.

## Plan close (when the last board row flips ✅)
The build session sets `NEXT: PLAN COMPLETE — run the master-plan skill` plus
an OWNER_TODO item; the closeout ritual is run by the master-plan skill, never
the build session.

## Amendment procedure (spec is wrong or must change)
Append to PROGRESS.md ## Amendments: `- A-P<NN>-<##> (<date>, affects <step ids>):
<what/why in <=3 lines>. Plan unchanged.` Deep rationale goes in the commit body.

## Budgets (checked by planning/tools/plan_check.py — run it to see the full table)
Enforced (FAIL, blocks handoff): PROGRESS.md <=32KB/400 lines · lines <=120
chars · 3 hot session entries · 6 fields <=240 chars each · plan file
<=96KB/1,200 lines · step <=40 lines · Facts <=15 bullets · Decisions <=20
live · SESSION_PROMPT.md <=3.5KB.
Advisory (WARN): <=30 steps/plan · amendments <=3 lines · decision one-liners
<=160 chars · archive shards 25 entries.

## Project-specific additions (hand-edit below this line only)
- {{none yet}}
```

## 9. planning/archive/ files

**SESSIONS-{{0001-0025}}.md** — rotated session entries, verbatim, oldest first,
exactly 25 per shard, then a new shard. Header:

```markdown

```

**PLAN-NN-CLOSEOUT.md** — generated at plan close, then immutable:

```markdown
<!-- PLANDOC:CLOSEOUT v1 — frozen at plan close. -->
# PLAN-{{NN}} {{slug}} — closeout ({{date}})
Final: {{N}}/{{N}} steps · sessions S{{aaaa}}–S{{bbbb}}

## Final board
{{the full board table, moved verbatim from PROGRESS.md}}

## Amendments ({{count}})
{{this plan's amendment lines, moved verbatim}}

## Decisions retired with this plan
{{decision lines that no longer bind future work}}

## Milestone rollup (what shipped)
| Milestone | Steps | Shipped |
|---|---|---|
| M1 {{title}} | S01–S06 ✅ | {{<=60 chars}} |
```

After the closeout is written, PROGRESS.md keeps: one Dashboard row, one line
under ## Completed plans linking here, and nothing else about this plan.

**PLAN-NN-PARKED.md** — written by the plan-park procedure (owner ruling to
set aside an incomplete plan — see the skill's conversion-rules.md): the full
board moved verbatim + resume notes. On resume, the board is rebuilt from this
file verbatim and the Dashboard row flips back to 🔵 ACTIVE.
